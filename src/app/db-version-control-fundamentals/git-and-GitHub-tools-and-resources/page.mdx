# Git and GitHub Tools And Resources

## Table of Contents

## Introduction

These resources and tools are designed to help you understand the fundamentals of version control using Git and GitHub. They cover essential concepts, practical applications, and best practices for managing code changes effectively.

Version control is the backbone of modern software development, enabling teams to collaborate efficiently, track changes over time, and maintain code quality. Git, as a distributed version control system, provides powerful capabilities for managing source code, while GitHub extends these capabilities with cloud-based collaboration features, project management tools, and automation workflows.

This comprehensive guide covers everything from basic Git operations to advanced workflows, GitHub integrations, and specialized tools for database version control. Whether you're a beginner learning your first Git commands or an experienced developer looking to optimize your workflow, these resources will help you master version control for your database and backend development projects.

## Git Fundamentals

### Core Concepts

#### Repository Structure

A Git repository consists of several key components that work together to track changes and manage versions:

```
.git/
‚îú‚îÄ‚îÄ HEAD                    # Points to current branch
‚îú‚îÄ‚îÄ config                  # Repository configuration
‚îú‚îÄ‚îÄ description            # Repository description
‚îú‚îÄ‚îÄ hooks/                 # Git hooks scripts
‚îú‚îÄ‚îÄ info/                  # Global excludes
‚îú‚îÄ‚îÄ objects/               # Git objects (commits, trees, blobs)
‚îú‚îÄ‚îÄ refs/                  # References (branches, tags)
‚îî‚îÄ‚îÄ logs/                  # Reflogs for references
```

#### The Three States

Git has three main states that your files can reside in:

1. **Working Directory**: Your local files where you make changes
2. **Staging Area (Index)**: Prepared changes ready to be committed
3. **Repository**: Committed changes stored in Git's database

```bash
# Working Directory ‚Üí Staging Area
git add filename.txt

# Staging Area ‚Üí Repository
git commit -m "Add new feature"

# Check status of all three areas
git status
```

#### Git Objects

Git stores data as objects in a key-value database:

- **Blob**: File contents
- **Tree**: Directory structure
- **Commit**: Snapshot with metadata
- **Tag**: Named reference to a commit

```bash
# View object type and content
git cat-file -t <object-hash>
git cat-file -p <object-hash>

# View repository structure
git ls-tree HEAD
```

### Basic Git Configuration

#### Global Configuration

```bash
# Set user information
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

# Set default editor
git config --global core.editor "code --wait"

# Set default branch name
git config --global init.defaultBranch main

# Configure line ending handling
git config --global core.autocrlf input  # Linux/macOS
git config --global core.autocrlf true   # Windows

# View all configuration
git config --list
```

#### Repository-specific Configuration

```bash
# Set configuration for current repository only
git config user.name "Work Name"
git config user.email "work.email@company.com"

# View current configuration
git config --show-origin --list
```

#### Useful Aliases

```bash
# Create helpful aliases
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.cm commit
git config --global alias.unstage 'reset HEAD --'

git config --global alias.plog "log --graph --pretty='format:%C(red)%d%C(reset) %C(yellow)%h%C(reset) %ar %C(green)%aN%C(reset) %s'"
```

## Git Commands Reference

### Essential Commands

#### Repository Management

```bash
# Initialize new repository
git init
git init --bare  # For server repositories

# Clone existing repository
git clone <url>
git clone <url> <directory-name>
git clone --depth 1 <url>  # Shallow clone

# Add remote repository
git remote add origin <url>
git remote add upstream <url>  # For forks

# View remotes
git remote -v
git remote show origin
```

#### Staging and Committing

```bash
# Add files to staging area
git add .                    # Add all files
git add *.js                 # Add all JavaScript files
git add src/                 # Add entire directory
git add -p                   # Interactive staging

# Remove files from staging
git reset HEAD <file>        # Unstage file
git reset                    # Unstage all files

# Commit changes
git commit -m "Commit message"
git commit -am "Add and commit"  # Stage and commit tracked files
git commit --amend           # Amend last commit
git commit --amend --no-edit # Amend without changing message
```

#### Branching and Merging

```bash
# Branch operations
git branch                   # List local branches
git branch -a                # List all branches
git branch <branch-name>     # Create new branch
git branch -d <branch-name>  # Delete branch
git branch -D <branch-name>  # Force delete branch

# Switch branches
git checkout <branch-name>
git checkout -b <branch-name>  # Create and switch
git switch <branch-name>       # Modern syntax
git switch -c <branch-name>    # Create and switch

# Merging
git merge <branch-name>
git merge --no-ff <branch-name>  # Force merge commit
git merge --squash <branch-name> # Squash merge
```

#### Viewing History

```bash
# Log commands
git log
git log --oneline
git log --graph --oneline --all
git log -p                   # Show patches
git log --stat               # Show file statistics
git log --since="2 weeks ago"
git log --author="John Doe"
git log --grep="bug fix"

# Show specific commit
git show <commit-hash>
git show HEAD~1              # Previous commit
git show <branch>:<file>     # File at specific commit

# Compare changes
git diff                     # Working directory vs staging
git diff --staged            # Staging vs last commit
git diff HEAD~1              # Working directory vs previous commit
git diff <branch1>..<branch2>
```

### Advanced Commands

#### Stashing

```bash
# Stash operations
git stash                    # Stash current changes
git stash save "message"     # Stash with message
git stash list               # List all stashes
git stash show               # Show stash contents
git stash apply              # Apply latest stash
git stash apply stash@{2}    # Apply specific stash
git stash pop                # Apply and remove stash
git stash drop               # Delete stash
git stash clear              # Delete all stashes

# Advanced stashing
git stash -u                 # Include untracked files
git stash --include-untracked
git stash -p                 # Interactive stashing
```

#### Rewriting History

```bash
# Interactive rebase
git rebase -i HEAD~3         # Rebase last 3 commits
git rebase -i <commit-hash>  # Rebase from specific commit

# Reset operations
git reset --soft HEAD~1      # Keep changes staged
git reset --mixed HEAD~1     # Keep changes unstaged (default)
git reset --hard HEAD~1      # Discard all changes

# Cherry-pick
git cherry-pick <commit-hash>
git cherry-pick <commit1>..<commit2>

# Revert
git revert <commit-hash>
git revert HEAD              # Revert last commit
git revert --no-commit <commit-hash>  # Revert without committing
```

## Git Workflows and Branching Strategies

### Git Flow

Git Flow is a branching model that defines specific branch types and their purposes:

```bash
# Install git-flow
# macOS: brew install git-flow
# Ubuntu: sudo apt-get install git-flow

# Initialize git-flow
git flow init

# Feature branches
git flow feature start feature-name
git flow feature finish feature-name

# Release branches
git flow release start 1.0.0
git flow release finish 1.0.0

# Hotfix branches
git flow hotfix start hotfix-name
git flow hotfix finish hotfix-name
```

#### Git Flow Branch Structure

```
main (production)
‚îú‚îÄ‚îÄ develop (development)
‚îÇ   ‚îú‚îÄ‚îÄ feature/user-authentication
‚îÇ   ‚îú‚îÄ‚îÄ feature/payment-integration
‚îÇ   ‚îî‚îÄ‚îÄ feature/admin-dashboard
‚îú‚îÄ‚îÄ release/1.2.0
‚îî‚îÄ‚îÄ hotfix/critical-bug-fix
```

### GitHub Flow

A simpler workflow suitable for continuous deployment:

```bash
# 1. Create feature branch from main
git checkout main
git pull origin main
git checkout -b feature/new-feature

# 2. Make changes and commit
git add .
git commit -m "Add new feature"

# 3. Push and create pull request
git push origin feature/new-feature

# 4. After review and merge, cleanup
git checkout main
git pull origin main
git branch -d feature/new-feature
```

### GitLab Flow

Combines Git Flow and GitHub Flow with environment branches:

```bash
# Environment-based workflow
main (production)
‚îú‚îÄ‚îÄ pre-production
‚îú‚îÄ‚îÄ staging
‚îî‚îÄ‚îÄ feature-branches
```

### Trunk-based Development

Short-lived branches with frequent integration:

```bash
# Daily workflow
git checkout main
git pull origin main
git checkout -b task/small-change
# Make small changes
git commit -m "Small incremental change"
git push origin task/small-change
# Create PR and merge same day
```

## GitHub Actions and CI/CD

### Basic Workflow Structure

```yaml
# .github/workflows/ci.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:
  NODE_VERSION: "18"
  PYTHON_VERSION: "3.9"

jobs:
  test:
    runs-on: ubuntu-latest

    strategy:
      matrix:
        node-version: [16, 18, 20]

    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Run linting
        run: npm run lint

      - name: Run tests
        run: npm test -- --coverage

      - name: Upload coverage reports
        uses: codecov/codecov-action@v3
        with:
          file: ./coverage/lcov.info
```

```yaml
# .github/workflows/advanced.yml
name: Advanced Workflow

on:
  push:
    branches: [main]
    tags: ["v*"]

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      version: ${{ steps.version.outputs.version }}

      - name: Generate version
        id: version
        run: |
          if [[ $GITHUB_REF == refs/tags/* ]]; then
            VERSION=${GITHUB_REF#refs/tags/}
          else
            VERSION=$(git describe --tags --always --dirty)
          fi
          echo "version=$VERSION" >> $GITHUB_OUTPUT

      - name: Deploy to production
        run: |
          echo "Deploying version ${{ needs.build.outputs.version }}"
```

## Advanced Git Techniques

### Git Hooks

Git hooks are scripts that run automatically when certain Git events occur:

#### Client-side Hooks

```bash
# .git/hooks/pre-commit
#!/bin/sh
# Run tests before commit
npm test
if [ $? -ne 0 ]; then
    echo "Tests failed. Commit aborted."
    exit 1
fi

# Run linting
npm run lint
if [ $? -ne 0 ]; then
    echo "Linting failed. Commit aborted."
    exit 1
fi
```

```bash
# .git/hooks/commit-msg
#!/bin/sh
# Validate commit message format
commit_regex='^(feat|fix|docs|style|refactor|test|chore)(\(.+\))?: .{1,50}'

if ! grep -qE "$commit_regex" "$1"; then
    echo "Invalid commit message format!"
    echo "Format: type(scope): description"
    echo "Types: feat, fix, docs, style, refactor, test, chore"
    exit 1
fi
```

#### Server-side Hooks

```bash
# hooks/pre-receive
#!/bin/sh
# Validate pushed commits
while read oldrev newrev refname; do
    # Check if branch is protected
    if [ "$refname" = "refs/heads/main" ]; then
        echo "Direct pushes to main branch are not allowed"
        exit 1
    fi
done
```

### Git Submodules

Managing external dependencies as Git repositories:

```bash
# Add submodule
git submodule add https://github.com/user/library.git libs/library

# Initialize submodules after cloning
git submodule init
git submodule update

# Or combine both commands
git submodule update --init --recursive

# Update submodule to latest
cd libs/library
git checkout main
git pull
cd ../..
git add libs/library
git commit -m "Update library submodule"

# Remove submodule
git submodule deinit libs/library
git rm libs/library
rm -rf .git/modules/libs/library
```

### Git Worktrees

Work on multiple branches simultaneously:

```bash
# Create new worktree
git worktree add ../feature-branch feature/new-feature

# List worktrees
git worktree list

# Remove worktree
git worktree remove ../feature-branch

# Prune deleted worktrees
git worktree prune
```

### Advanced Rebase Techniques

#### Interactive Rebase Operations

```bash
# Interactive rebase with various operations
git rebase -i HEAD~5

# Available operations in editor:
# pick = use commit
# reword = use commit, but edit message
# edit = use commit, but stop for amending
# squash = use commit, but meld into previous commit
# fixup = like squash, but discard commit message
# exec = run command
# drop = remove commit
```

#### Rebase onto Different Base

```bash
# Rebase feature branch onto updated main
git checkout feature/new-feature
git rebase origin/main

# Rebase with strategy
git rebase -X theirs origin/main  # Prefer their changes
git rebase -X ours origin/main    # Prefer our changes

# Preserve merge commits
git rebase --preserve-merges origin/main
```

```yaml
        run: |
          CHANGED_FILES=$(git diff --name-only origin/${{ github.base_ref }}...HEAD | wc -l)
          CHANGED_LINES=$(git diff --stat origin/${{ github.base_ref }}...HEAD | tail -1 | awk '{print $4}' | sed 's/[^0-9]*//g')

          if [ "$CHANGED_FILES" -gt 20 ]; then
            echo "::warning::PR contains $CHANGED_FILES files. Consider breaking into smaller PRs."
          fi

          if [ "$CHANGED_LINES" -gt 500 ]; then
            echo "::warning::PR contains $CHANGED_LINES changed lines. Consider breaking into smaller PRs."
          fi

      - name: Check commit messages
        run: |
          git log --format="%s" origin/${{ github.base_ref }}...HEAD | while read msg; do
            if ! echo "$msg" | grep -qE '^(feat|fix|docs|style|refactor|test|chore)(\(.+\))?: .+'; then
              echo "::error::Invalid commit message: $msg"
              exit 1
            fi
          done
```

### Backup and Disaster Recovery

#### Automated Backup Strategy

```bash
#!/bin/bash
# backup-repos.sh
BACKUP_DIR="/backups/git-repos"
DATE=$(date +%Y%m%d_%H%M%S)

# Create backup directory
mkdir -p "$BACKUP_DIR/$DATE"

# Backup all repositories
find /opt/git -name "*.git" -type d | while read repo; do
    repo_name=$(basename "$repo" .git)
    echo "Backing up $repo_name..."

    # Create bundle with all refs
    git --git-dir="$repo" bundle create "$BACKUP_DIR/$DATE/$repo_name.bundle" --all

    # Backup repository metadata
    tar -czf "$BACKUP_DIR/$DATE/$repo_name-metadata.tar.gz" -C "$repo" hooks config description
done

# Clean old backups (keep last 30 days)
find "$BACKUP_DIR" -type d -mtime +30 -exec rm -rf {} \;
```

```bash
# Restore from bundle
git clone backup.bundle restored-repo
```
